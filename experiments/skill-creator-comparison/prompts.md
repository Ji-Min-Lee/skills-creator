# Skill Creator 비교 프롬프트

같은 프롬프트를 Claude creator와 Codex creator에 각각 그대로 입력해 결과를 비교합니다.

---

## 1. Skill 생성 프로세스 비교

```text
다음 요구를 만족하는 새로운 skill을 만든다고 가정해줘.

목표:
- 사용자가 CSV나 XLSX 매출 데이터를 주면
- 월별 매출 요약, 이상치 탐지, 간단한 시각화, 핵심 인사이트 요약을 해주는 skill을 만들고 싶다.

이 상황에서 너의 권장 방식대로:
1. 어떤 질문으로 요구사항을 구체화할지
2. 어떤 순서로 skill을 설계할지
3. 초안을 만들기 전 무엇을 먼저 결정할지
4. 새 skill 생성 절차를 단계별로 제시해줘

가능하면 “왜 그 순서가 좋은지”도 함께 설명해줘.
```

## 2. 권장 skill 구조 비교

```text
다음 용도의 skill을 만든다고 가정해줘.

용도:
- PDF 계약서를 읽고
- 핵심 조항 추출
- 위험 조항 표시
- 요약 리포트 생성

이 skill에 대해 권장하는 폴더 구조를 제안해줘.
반드시 아래를 포함해서 설명해줘:
- SKILL.md
- scripts/
- references/
- assets/
- agents/
- 그 외 파일이 필요하다면 무엇인지

또한 각 파일/폴더에 어떤 내용을 넣어야 하는지, 무엇은 넣지 말아야 하는지도 설명해줘.
```

## 3. SKILL.md 작성 철학 비교

```text
새로운 skill의 SKILL.md를 잘 쓰기 위한 작성 원칙을 설명해줘.

특히 아래를 비교 가능하게 답해줘:
1. frontmatter description은 어떻게 써야 하는가
2. body에는 무엇을 넣어야 하는가
3. examples는 어느 정도까지 넣는가
4. 강한 규칙(MUST/ALWAYS)과 유연한 가이드의 균형은 어떻게 잡는가
5. 긴 설명과 짧은 설명 중 무엇을 선호하는가

추상론 말고, 실제로 좋은 skill 문서를 쓰기 위한 구체적인 원칙 형태로 답해줘.
```

## 4. Context 관리 / progressive disclosure 비교

```text
skill이 너무 길어져서 context를 많이 먹는 상황을 가정하자.

예시:
- 하나의 skill이 AWS, GCP, Azure 세 가지 배포 방식을 모두 지원해야 한다.
- 관련 예시, API 설명, 운영 규칙도 많다.

이때 너라면:
1. 어떤 내용을 SKILL.md 본문에 남기고
2. 어떤 내용을 references/로 분리하고
3. 어떤 기준으로 파일을 나누며
4. 사용 중 필요한 정보만 읽히도록 어떻게 설계할지

구체적인 구조와 판단 기준을 설명해줘.
```

## 5. 테스트/검증 방법 비교

```text
새로 만든 skill이 실제로 잘 동작하는지 검증하고 싶다.

예시 skill:
- 사용자의 회의 녹취 텍스트를 받아
- action items, decisions, risks를 구조화해서 정리하는 skill

이 skill을 검증하기 위해:
1. 어떤 테스트 프롬프트를 만들지
2. 테스트 케이스는 몇 개 정도부터 시작할지
3. 어떤 실패 케이스를 꼭 포함할지
4. 실행 전/후 어떤 검증을 할지
5. 수동 검토와 자동 검증을 어떻게 나눌지

너의 권장 검증 절차를 단계별로 제안해줘.
```

## 6. 성능 평가 방법 비교

```text
skill 성능을 체계적으로 평가하는 방법을 알려줘.

예시 skill:
- 사용자의 messy한 매출 데이터를 받아
- 정리된 요약과 차트를 만드는 skill

이 skill에 대해:
1. 정성 평가를 어떻게 할지
2. 정량 평가를 어떻게 할지
3. 어떤 지표를 쓸지
4. baseline과 비교할 필요가 있는지
5. 사용자 피드백을 어떤 방식으로 수집/반영할지

실험 가능한 평가 방법으로 제시해줘.
```

## 7. 성능 최적화 방법 비교

```text
어떤 skill의 첫 번째 버전이 기대보다 별로였다고 하자.

문제 예시:
- 출력이 장황하다
- 중요한 정보가 빠진다
- 테스트 프롬프트 몇 개에는 맞지만 일반화가 잘 안 된다
- 실행 중 불필요한 중간 작업을 많이 한다

이 상황에서 skill 자체를 어떻게 개선해 나갈지 설명해줘.
반드시 아래를 포함해줘:
1. 실패 원인 분석 방법
2. 과적합 없이 일반화하는 방법
3. SKILL.md를 어떻게 줄이거나 바꿀지
4. scripts/references/assets 추가 여부 판단 기준
5. 반복 개선 루프 설계
```

## 8. 트리거링 최적화 비교

```text
좋은 skill을 만들었는데 잘 트리거되지 않는다고 가정하자.

이때:
1. description을 어떻게 고칠지
2. 어떤 식의 should-trigger / should-not-trigger 예시를 만들지
3. trigger 품질을 어떻게 평가할지
4. 과도한 오탐과 미탐을 어떻게 줄일지

설명해줘.

예시로:
- 사용자가 문서 요약, 계약서 검토, 조항 비교, 위험 포인트 찾기를 요청할 때 잘 트리거되어야 하는 legal-doc-review skill
을 기준으로 답해줘.
```

## 9. 자동화/도구 활용 비교

```text
skill을 사람이 수작업으로 만드는 대신, 도구와 스크립트를 적극 활용해 만들고 싶다.

새로운 skill을 만드는 전체 과정에서:
1. 어떤 부분을 자동화할 수 있는지
2. scaffold 생성이 필요한지
3. validation 도구가 필요한지
4. benchmark/eval 도구가 필요한지
5. packaging이나 배포 준비가 필요한지

너의 이상적인 도구 기반 workflow를 제안해줘.
```

## 10. 기존 skill 개선 비교

```text
이미 존재하는 skill을 개선하려고 한다고 가정하자.

상황:
- 기존 skill은 동작은 하지만 설명이 길고
- 구조가 지저분하고
- 테스트가 부족하고
- 실제 사용자 요청에서 자주 빗나간다.

이 경우:
1. 기존 skill을 어떻게 진단할지
2. 새 skill 생성과 무엇이 다른지
3. 어떤 것은 유지하고 어떤 것은 바꿀지
4. baseline 비교는 무엇을 기준으로 할지
5. 안전하게 개선하는 절차는 무엇인지

단계별 개선 전략으로 설명해줘.
```
